Client360 · Design Specification

Application Shell & Design System

A single, capability-honest interface for the whole platform — one shell, one component language, one type and color system — proposed for approval before any template is migrated.

Release 0.9.12 — UI Consolidation Status Design spec · pre-approval Scope Frontend only · no backend / auth change Stack Server-rendered Jinja2 + CSS (unchanged)
01

What this system is for

Client360 is operated all day by advisors, tax preparers, operations, compliance, and administrators. The interface is an instrument, not a brochure — dense where the work is dense, calm everywhere else, and always honest about what a given user is allowed to see.

Three commitments

  • Summary before detail. Every screen leads with the numbers that drive a decision, then the rows.
  • State reads at a glance. Severity, status, and SLA are encoded as color and shape — a stripe, a pill, a tick — not just text.
  • Names, never raw IDs. People, households, organizations, and plans are named, clickable links; IDs appear only as quiet monospace metadata.

What does not change

  • All authorization and record-scope rules — the UI only reflects them, and the capability-filtered nav makes them more visible.
  • Routes, services, and the data model.
  • The server-rendered Jinja2 + CSS stack. No SPA, no second framework, no build step.
02

Color

A cool-slate neutral scale, biased a few degrees toward the brand so it reads as chosen rather than default. The primary brand color — a deep teal, #0E6E63 — carries Client360's identity and every interaction: primary buttons, active navigation, links, focus states, and key highlights. The four semantic colors are an independent system and never borrow the brand hue — Info is now its own understated steel-blue so "in progress" never reads as a brand highlight.

Nav is grouped by the job to be done and every item declares the capability it requires. Items a user cannot access are hidden — not shown-then-403. Error & denial states (replacing raw JSON)

403 · NOT AUTHORIZED

You don't have access to this screen

Firm-wide dashboards require the record.read_all capability. Your work lives under My Work.

404 · NOT FOUND

That record isn't here

It may have been removed, or it's outside the organizations you're assigned to.

Out-of-scope records return the same 404 as truly-missing ones — existence is never disclosed. The styling changes; the authorization behavior does not.

09

How it ships

Additive and reversible — templates and CSS only. No schema, no route or service signature changes. Phased per the established cadence, each phase committed separately with a stop-for-review.

PhaseScopeEffort
P0 FoundationTokenized app.css, Jinja component macros, shared render helper + context processor, styled error pages. Proven on 2 pilot screens.3–4 d
P1 ShellNew base.html: left nav (capability-filtered) + header + layout. Migrate the ~16 screens already on base.html.3–4 d
P2 Core screensConvert standalone templates — dashboard, work, households, people, tasks, activities, relationships, admin, M365 — onto the shell.5–7 d
P3 De-raw HTMLMove matches, search, people & peers from Python-built HTML into templates; replace all raw-ID links with named links.3–4 d
P4 InteractionTables (sort/paginate), search & filtering, form validation, empty/loading/success/error states.3–5 d
P5 Portal + RCAlign the portal shell to shared tokens; accessibility/responsive QA; authorization & record-scope regression across all personas; RC12 validation.3–4 d

Total ≈ 20–28 dev-days across 6 phases. Interaction states default to server-side (post-redirect-get, inline re-render). A tiny progressive-enhancement layer for async filtering is an optional, separately-approved decision — the default honors "no second frontend framework."

What approving this spec authorizes

The color and type systems, the shell (left nav + header), the capability-gated navigation model, and the component vocabulary in §6. On approval I begin Phase 0 only — the design foundation and two pilot screens — and stop for review before any broad template migration.
